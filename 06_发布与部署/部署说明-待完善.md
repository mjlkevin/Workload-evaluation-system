# 部署说明

## 当前状态

- 仓库根目录提供 **`docker-compose.yml`** 与 **`docker/Dockerfile.api`**，主前端提供 **`ui/V0_SAAS/Dockerfile`**，用于最小可复现部署（API + Next）。
- 本地开发仍以 `npm run dev:api` 与 `npm run dev:web`（`ui/V0_SAAS`）为主。

## 云部署流程（简版）

1. **选型与采购**
   - 云主机建议：Ubuntu 22.04，`2C4G` 起步。
   - 磁盘建议：`60G SSD`（推荐 `80G`，便于日志与备份）。
   - 域名建议提前采购并处理备案（按地区要求）。
2. **服务器初始化**
   - 创建普通运维用户。
   - 禁用密码登录，仅保留 SSH Key。
   - 仅开放 `22/80/443`，不要将 `3000/3001` 暴露公网。
   - 安装 Docker 与 Docker Compose。
3. **项目上云**
   - 拉取代码，准备生产 `.env`（`JWT_SECRET`、`KIMI_API_KEY` 等）。
   - 使用现有 `docker-compose.yml` 启动 API + Web。
4. **反向代理与 HTTPS**
   - Nginx/Caddy 反向代理到 `web:3001`。
   - `/api` 经前端代理到 `api:3000`。
   - 配置 Let's Encrypt 证书。
5. **上线验收**
   - 健康检查：`/api/v1/health`。
   - 主链路冒烟：登录、版本流转、导出、团队协作。
   - 观察 CPU/内存/磁盘与容器日志。
6. **运维基线**
   - 建立发布脚本（拉代码、重建、重启、回滚）。
   - 建立故障处理 SOP（证书、磁盘、服务异常）。

## Docker Compose（推荐入门）

在仓库根目录：

```bash
export JWT_SECRET="$(openssl rand -hex 32)"
docker compose up --build
```

- Web：`http://localhost:3001`
- API 健康检查：`http://localhost:3000/api/v1/health`
- **`config/`** 目录通过卷挂载到 API 容器，持久化用户、版本、团队等文件型数据；首次部署后请确认目录权限。

Next 通过环境变量 **`WORKLOAD_API_ORIGIN`** 将浏览器请求 `/api/*` 反代到 API（见 `ui/V0_SAAS/next.config.mjs`）。Compose 内默认 `http://api:3000`。

## 建议的生产形态（扩展）

1. **API**：Node 运行 `apps/api` 构建产物，暴露 `PORT`（默认 3000）。
2. **Web**：`ui/V0_SAAS` 构建后 `next start`，前置 Nginx 可选。
3. **持久化目录**（需挂载卷）：
   - `config/templates`、`config/rules`
   - `config/auth`、`config/versions`、`config/teams`
   - 导出文件目录（与运行时配置一致）
4. **环境变量**：根目录 `.env.example` 与 `docs/ENVIRONMENT.md`。
5. **TLS**：由 Nginx 或负载均衡终止 HTTPS。

## 首次部署检查清单

- [ ] `JWT_SECRET` 已更换为强随机值  
- [ ] `KIMI_API_KEY` 等密钥通过环境变量注入，未写入镜像层  
- [ ] 导出目录磁盘空间与清理策略（TTL）  
- [ ] `config/*` 备份策略  

## Beta 数据治理（单实例、低并发前提）

在不接入 DB 的前提下，Beta 期间必须补齐：

1. **每日自动备份 `config/*`**  
   - 命令：`npm run ops:backup:config`
2. **数据完整性校验（启动/定时）**  
   - 启动校验：API 启动时默认执行（可由 `CONFIG_INTEGRITY_ON_STARTUP=false` 关闭）
   - 手工校验：`npm run ops:check:config`
   - 修复模式：`npm run ops:check:config:repair`
3. **异常/修复日志沉淀**  
   - 日志文件：`logs/data-anomaly-repair.log`

### 定时任务示例（crontab）

```bash
# 每天 02:10 备份 config
10 2 * * * cd /path/to/WorkEvolutionSys && npm run ops:backup:config

# 每天 02:20 做完整性校验
20 2 * * * cd /path/to/WorkEvolutionSys && npm run ops:check:config
```

> 说明：数据库接入放在第三轮，且仅在迁移触发器满足后推进。

## 前端 E2E（Playwright）

在 **`ui/V0_SAAS`** 安装依赖后安装浏览器内核：

```bash
cd ui/V0_SAAS && npm ci && npx playwright install chromium
```

另开终端启动 API（3000）与 Next（3001），然后在 `ui/V0_SAAS` 执行：

```bash
npm run test:e2e
```

或在仓库根目录：

```bash
npm run test:e2e:web
```

带 `/api/v1/health` 的用例默认跳过；需要同时拉起 API（3000）与 Next（3001）时，执行：

`E2E_WITH_API=1 npm run test:e2e`

## 最小 CI（已接入）

- 工作流：`.github/workflows/minimal-ci.yml`
- 触发：`push main`、`pull_request`
- 默认检查项：
  - `npm run build:api`
  - `npm run test:modules`
  - `npm run test:rules`
  - `npm run test:integration`
  - `npm run build:web`
- E2E 建议先保持非阻塞（夜间任务或手工触发），避免首发阶段阻塞交付效率。
