# 三轮产品完整闭环需求输入 V1

> 对应计划待办：`r3-product-completion-input`  
> 目标：为三轮迭代（产品完整度提升）提供可立项的需求池、验收标准与依赖清单。  
> 前置依赖：首发 MVP 稳定 + 二轮数据层/集成骨架完成。

## 1. 三轮目标

- 完成评审链路、WBS、标准 PLM 的产品闭环。
- 提升跨模块一致性（状态、文案、流程、权限反馈）。
- 将“可演示”能力升级为“可运营”能力。

## 2. 需求池（候选史诗）

| 史诗ID | 名称 | 业务价值 | 优先级 | 依赖 | 验收口径（摘要） |
|---|---|---|---|---|---|
| EPIC-R3-001 | 评审全链路闭环 | 评审效率与可追溯性提升 | 高 | 团队协同 P0 + 二轮数据层 | 支持评审模板、状态流转、评论追踪、责任人可视化 |
| EPIC-R3-002 | WBS 产品化闭环 | 从派生视图升级为可运营任务面板 | 高 | WBS 二轮增强 | 支持任务编辑、依赖关系、状态推进、关联版本追溯 |
| EPIC-R3-003 | 标准 PLM 预置策略落地 | 统一方案模板与组织执行口径 | 中高 | 产品策略评审结论 | 至少 1 套标准策略可配置、可应用、可回溯 |
| EPIC-R3-004 | 跨模块体验一致性治理 | 降低学习成本与误操作 | 中 | 首发与二轮页面完成 | 统一状态机文案、按钮行为、错误提示与空态设计 |
| EPIC-R3-005 | 运营与审计视图补齐 | 管理者可见性增强 | 中 | 数据层聚合能力 | 增加关键看板（评审进度、任务完成率、异常分布） |
| EPIC-R3-006 | 数据库接入（触发式） | 支撑扩容与高审计需求 | 条件触发 | 迁移触发器成立 | 完成迁移评审后接入，不破坏 repository 抽象 |

## 3. 详细需求条目（立项输入）

### 3.1 评审闭环（EPIC-R3-001）

- REQ-R3-REV-001：评审模板配置（标题、检查项、必填结论）。
- REQ-R3-REV-002：评审状态机标准化（open/in_review/closed/rejected）。
- REQ-R3-REV-003：评论线程支持 @对象 与状态联动。
- REQ-R3-REV-004：评审审计字段补齐（操作人、时间、动作、备注）。

### 3.2 WBS 产品化（EPIC-R3-002）

- REQ-R3-WBS-001：任务可编辑字段（负责人、起止时间、状态、备注）。
- REQ-R3-WBS-002：任务与版本双向追溯（global 与子版本映射）。
- REQ-R3-WBS-003：关键路径视图（至少支持简单依赖关系展示）。
- REQ-R3-WBS-004：空态/异常态/冲突态统一反馈。

### 3.3 标准 PLM（EPIC-R3-003）

- REQ-R3-PLM-001：预置策略清单与适用条件定义。
- REQ-R3-PLM-002：策略应用记录（谁、何时、应用到哪个方案）。
- REQ-R3-PLM-003：策略版本治理（新增、废弃、兼容说明）。

### 3.4 跨模块一致性（EPIC-R3-004）

- REQ-R3-UX-001：版本状态文案统一（检入/检出/评审/归档）。
- REQ-R3-UX-002：权限拒绝提示统一（401/403 的用户态反馈）。
- REQ-R3-UX-003：表单脏数据离开提醒统一（防误操作）。

### 3.5 数据库接入（EPIC-R3-006，触发式）

- REQ-R3-DB-001：迁移触发器达成后方可立项（并发冲突、性能阈值、扩容、审计追溯）。
- REQ-R3-DB-002：保持 repository 边界不变，业务层不感知存储介质切换。
- REQ-R3-DB-003：提供双轨回滚策略（文件存储可回退）。

## 4. 立项门槛（DoR）

- 需求具备唯一 ID、业务目标、验收标准、依赖说明。
- 至少包含 1 条可测试主场景与 1 条异常场景。
- 明确归属轮次，禁止“首发范围外”需求直接插入首发计划。
- 涉及 DB 的需求必须附“触发器证据”与“迁移回滚方案”。

## 5. 评审与排期建议

- 评审顺序：`EPIC-R3-001` -> `EPIC-R3-002` -> `EPIC-R3-003` -> `EPIC-R3-004/005`
- 建议拆分方式：每个史诗按“接口契约 -> 页面交互 -> 回归测试 -> 文档”四段推进。
- 建议输出：三轮立项评审会后，形成 `Freeze List v3` 与对应迭代排期表。
