# 团队协同（P0）需求评审结论（默认方案，可开工）

> 说明：你回复“继续”但未对评审问题逐项拍板，因此本文件采用 **最小可交付/低风险** 默认方案，用于直接进入开发。后续如需调整，按每节“可选变体”替换即可。

## 1. P0 目标（本期必须达成）

- 建立团队组织模型（团队、成员、角色）。
- `部门负责人` 可查看本团队下全部评估方案（总方案 GL- 视角）。
- 团队成员可围绕同一总方案开展评审与评论协作。
- 关键动作留痕，便于回溯与责任界定。

## 2. P0 不做（明确边界）

- 不做字段级复杂 RBAC 引擎。
- 不做任务分派/截止时间/通知（@提醒、消息推送）——归入 P1。
- 不做跨团队共享/外部协作（对外分享链接）——归入 P2。
- 不做 BI 看板与绩效统计——归入 P2。

## 3. 核心定义（P0拍板项：默认）

### 3.1 团队与总方案（GL-）归属

- **默认结论**：`手动关联`  
  - 负责人在总方案上手动选择“归属团队”（可后续调整）。
- **理由**：避免“创建人可能属于多个团队/未来跨部门协作”的歧义；对现存数据改动最小。
- **可选变体**：
  - `自动归属`：总方案创建人默认归属其主团队（需要定义主团队）。
  - `自动 + 可改`：先自动归属，再允许负责人调整。

### 3.2 团队内可见范围

- **默认结论**：`团队成员可见团队内全部方案（只读），评论/评审按角色限制`。
- **可选变体**：
  - 仅可见“指派给自己”的方案（负责人可见全部）。
  - 可见全部但敏感字段遮罩（成本/报价）。

### 3.3 评审/评论对象粒度

- **默认结论**：`仅总方案（GL-）级别`。
- **理由**：最小闭环，避免 P0 同时兼容多个子版本对象与跳转入口。
- **可选变体**：总方案 + 子版本（PG/RI/DV/RS）都可评论（P1）。

### 3.4 评审流程

- **默认结论**：`open/closed 两态评审 + 负责人关闭 + 留痕`。
- **可选变体**：
  - 只有评论（无评审状态）。
  - 带任务分派（P1）。

### 3.5 角色与权限最小映射

- **默认结论**：`manager/member 两级落地 + 保留 role 字段可扩展`  
  - `部门负责人` = manager（管团队、关评审、看全部）
  - `实施顾问/售前顾问/销售顾问` = member（可查看、可评论、可参与评审）

### 3.6 留痕范围（审计）

- **默认结论**：仅关键动作留痕：
  - 创建团队 / 修改成员
  - 创建评审 / 关闭评审
  - 发表评论

### 3.7 存储方式（与现状一致）

- **默认结论**：`文件化 JSON`，但 repository 抽象预留未来切 SQLite（架构先行）。

## 4. 关键页面/入口（P0）

- 团队管理页（负责人）：团队信息、成员列表、角色调整。
- 团队方案总览页（负责人 + 成员）：GL- 列表（状态/更新时间/负责人/评审状态）。
- 总方案详情页新增“评审/评论”区域：
  - 显示当前评审状态（open/closed）
  - 评论列表 + 发表评论
  - manager 可关闭评审

## 5. 验收标准（P0）

- 能创建团队并配置成员与四类角色。
- 部门负责人能看到团队内全部 GL- 方案列表。
- 团队成员能在同一 GL- 下发起/参与评审与评论。
- 非本团队成员无法访问团队方案与评审内容。
- 系统保留关键动作的最小审计记录，可追溯到人/时间/对象。

## 6. P1/P2 预留

- P1：评论线程/@提醒、任务分派、子版本级评论、通知机制。
- P2：协同效率看板、跨团队协作/分享、权限细化与组织架构对接（部门树/SSO）。
