# R2 数据层演进与集成启动方案（首版）

> 对应计划待办：`r2-data-integration`  
> 目标：在不影响首发 MVP 的前提下，启动“文件存储 -> 目标数据层”的演进与外部集成骨架。

## 1. 启动边界

- 本阶段只做样板链路，不做全量迁移。
- 仍保持当前文件存储可运行，迁移采用双轨策略（可回滚）。
- 优先覆盖高价值链路：`versions`、`teams`、`wbs` 派生依赖信息。
- 当前不接 DB：部署前提为单实例、并发不高，核心目标是验证流程与产品价值。

## 2. 演进分层

### 2.1 存储抽象层

- 将业务域读写入口收敛为“仓储适配层”概念，避免调用方直接依赖文件结构。
- 现阶段保留文件实现，后续可平滑替换为 SQLite 或其他持久化实现。

### 2.2 数据模型收敛

- 统一关键实体主键与审计字段：
  - `id` / `createdAt` / `updatedAt`
  - `ownerUserId`（租户隔离最小字段）
  - `version`（并发控制预留）
- 强化版本关联字段一致性（尤其是 `global` 与子模块版本映射）。

### 2.3 WBS 增强起步

- 当前已将 WBS 返回结构扩展为“可追溯派生任务”：
  - `moduleKey`
  - `linkedVersionCode`
  - `sourceGlobalVersionCode`
  - `sourceGlobalRecordId`
  - `isDerived`
- 该结构用于二轮中接入持久化任务模型与运营看板。

## 3. 外部集成骨架（R2）

- 先定义契约，再接入实现：
  - 输入侧：版本/评估摘要查询接口
  - 输出侧：任务状态与评审结果回写接口
- 首轮只要求可联调，不要求高并发与全量补偿机制。

## 4. 风险与回滚

- 风险：迁移期间出现字段口径不一致导致前后端解析失败。
- 措施：
  - 迁移变更必须附带契约校验（OpenAPI + 用例）。
  - 保留文件存储兜底开关，异常时可切回。
  - 二轮结束前至少完成一次全链路回归（versions/team/wbs）。

## 5. 二轮里程碑建议

- M1：完成样板链路设计评审（版本、团队、WBS）。
- M2：完成 1~2 条核心链路迁移与回归。
- M3：输出“可继续迁移清单”与“三轮产品化输入”。

## 6. 三轮 DB 立项触发器（计划前置）

- 每周出现 >= 2 次需要人工修复的并发写冲突。
- 核心查询 P95 持续超过团队阈值。
- 明确需要多实例部署/容器扩容。
- 审计、统计、权限追溯需求显著增加。

> 触发器满足后再推进数据库接入；迁移过程中必须保持 repository 边界不变，业务层不直接依赖 JSON 文件结构。
