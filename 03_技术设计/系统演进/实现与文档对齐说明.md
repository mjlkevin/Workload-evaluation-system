# 实现与文档对齐说明（以代码为准）

> 本文档描述**当前仓库实现**与早期 V2 设计文档（轻量「仅计算导出」、无登录、三页面前端等）之间的差异，作为技术事实的单一补充来源。  
> V2 文档仍保留历史决策与评审结论；涉及「现状行为」请以本文 + `apps/api/src/app.ts`、`apps/api/src/routes/index.ts` 为准；前端统一以 `ui/V0_SAAS` 为准（【历史说明，已下线】`apps/web`）。

## 1. 一句话摘要

系统在保留「模板 + 规则 + 计算 + 导出」核心的基础上，已演进为：**多页签业务工作台 + JWT 登录 + 文件型用户/推荐码/版本记录 + 浏览器端草稿（按用户隔离）+ 需求 Excel 智能解析（Kimi + 规则回退）**。

## 2. 技术栈（实现）

| 维度 | V2 文档常见表述 | 当前实现 |
|------|-----------------|----------|
| 后端框架 | NestJS 等 | **Express**（`createApp` 于 `apps/api/src/app.ts`，路由于 `apps/api/src/routes/index.ts`） |
| 前端 UI 库 | Element Plus | **当前主界面**：Next.js（`ui/V0_SAAS`） |
| 状态管理 | Pinia | **当前实现**：React 组件状态（按页面拆分） |
| 鉴权 | 可选 / `X-Role` | **JWT `Authorization: Bearer`**；业务接口普遍要求登录；**不再信任客户端 `X-Role`** |
| 持久化 | 模板/规则/导出文件 | 另增 **`config/auth/users.json`、`config/auth/invite-codes.json`、`config/versions/records.json`、`config/versions/version-code-rules.json`、`config/teams/store.json`**（团队协同 + 系统管理） |

## 4. 角色模型（实现）

- 用户表字段角色为 **`admin` | `user`**（首注册用户为 `admin`）。
- 接口层将 **`user` 映射为与 V2「operator」等价的可估算角色**（`resolveApiRoleFromUser`）：普通用户可进行计算、导出、版本读写等业务；**仅 `admin` 可**模板/规则导入、用户与推荐码管理。

## 5. 版本与前端草稿（实现）

- **浏览器 `localStorage`**：实施评估、资源、总方案、需求、开发评估等草稿 **按 `userId` 分 key**；登出清理。
- **服务端 `config/versions/records.json`**：统一记录多类版本（实施/资源/需求/开发/总方案等），与前端保存联动；删除时校验是否被总方案引用。
- **审计字段**：每条版本记录含 **创建人**（`createdByUserId` / `createdByUsername`）与 **最近修改人**（`updatedByUserId` / `updatedByUsername`）；创建时二者相同，检出/检入/撤销、状态变更、升版、强制解锁等会更新 `updatedAt` 与修改人；读盘时 `migrateVersionRecord` 为缺字段的旧数据回填。各模块「单据列表」表格展示创建人、修改人（用户名）。
- 版本号编码由系统管理规则驱动（`config/versions/version-code-rules.json`）；仅 `active` 规则可用于创建版本。

## 6. 前端信息架构（实现）

- **当前主产品（Next，`ui/V0_SAAS`）**：按 `app/` 路由组织（如 `dashboard/assessment`、`dashboard/system-management` 等），对接 `/api/v1/*`。
- 与 V2「模板选择 / 录入 / 结果」三页静态拆分不一致；**三页逻辑已融合进「实施评估」工作台 + 结果区**（当前 Next 语境）。
- **实施评估 · 版本交互（产品规则）**：
  - 主参数区「当前生效版本」下拉**仅展示**当前用户实施评估版本中按 `updatedAt` 最新的一条（与「只认最新生效」一致）。
  - 全量历史在「版本历史」弹窗；需基于旧快照继续工作时，选中历史行后使用 **「按历史版本创建新版」**：深拷贝该条 `payload` 调用 `POST /api/v1/versions` 生成**新编码**（版本号只增不减）。
  - 嵌入 URL 若带非当前生效的 `version` 查询参数，前端会提示并仍加载当前生效版本（与上述规则一致）。
- **实施评估 / 开发评估 · 项目名称**：与仪表盘总方案 `projectName` 联动或手填，随版本 payload 持久化；导出实施评估时使用当前项目名称作为导出展示名。
- **实施评估 · 导出 Excel**：以 `01_需求管理/原始需求/实施评估RR/金蝶AI星空-实施人天估算-R202602-V1.0（0303版本）.xlsx` 为版式原型；`POST /api/v1/estimates/calculate-and-export`（及导出类请求体）可带 `selectedSheet`、`selectedCloudNames`。表体仅含当前工作表内已勾选且（若传云产品列表）命中云产品筛选的行；底部汇总区写入服务端计算后的数值（与导出行范围一致的重算结果），避免依赖可能损坏的单元格公式。写出文件时使用 **ExcelJS** 应用与参考表一致的填充色与边框（社区版 `xlsx` 仍用于读取参考模板）。**导出列范围**：以表头「小计」列为右边界（一般为 J 列），不包含参考表中「R202505版本对标」等 K 列及以右内容；**页脚**：在导出列范围内剔除尾部全空行，避免页脚末尾多余空白行。
- **通用弹窗**：`components/ui/dialog.tsx` 支持顶部空白区拖拽位移（全局生效）。

## 7. 环境变量（实现摘录）

| 变量 | 用途 |
|------|------|
| `PORT` | API 端口，默认 `3000` |
| `JWT_SECRET` | JWT 签名密钥 |
| `JWT_EXPIRES_IN` | Token 有效期，默认 `8h` |
| `KIMI_API_KEY` | 需求 Excel 解析（Moonshot/Kimi） |
| `KIMI_MODEL` | 模型名，默认 `kimi-k2.5` |
| `KIMI_API_BASE_URL` | API 基址，默认官方地址 |

加载顺序：优先 `.env.local`，再 `.env`（见 `apps/api` 环境加载逻辑）。
