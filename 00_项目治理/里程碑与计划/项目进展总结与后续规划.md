# 项目进展总结与后续规划

> 更新说明：本文与 `README.md`、`03_技术设计/系统演进/实现与文档对齐说明.md`、`对话流程总结/对话流程与里程碑总览.md` 互为补充；**技术事实以代码与对齐说明为准**。Agent 高层 `/api/v1/agent/*` 为规划项，**当前主线未挂载**（见对齐说明 §3.2）。

## 一、当前进展总览（截至 2026-03）

### 1. 产品形态

系统已从早期「单评估页 + 计算导出」演进为 **多页签评估工作台**：

| 模块 | 状态 | 说明 |
|------|------|------|
| 总览 | 可用 | 评估方案列表、新建方案向导、方案预览（关联关系 + 基本分析 + 子评估摘要） |
| 需求 | 可用 | 多区块表单、Excel 上传、Kimi + 规则回退解析、独立 `RI-` 版本 |
| 实施评估 | 可用 | 模板勾选、预置模式、自定义人天、云产品折叠、临时保存 `PG-` 版本 |
| 开发评估 | 可用 | 开发人天表、`DV-` 版本、可关联实施评估 SKU |
| 资源人天及成本 | 可用 | 可编辑表、图表、`RS-` 版本、模块标签选择、差旅列开关 |
| WBS | 只读 MVP | `GET /api/v1/wbs` 基于最新 `GL-` 派生占位任务行；持久化与编辑待迭代 |
| 评审 | 部分可用 | 团队维度评审/评论见 `/api/v1/teams/.../reviews`；前端页面与闭环体验继续迭代 |
| 用户管理 / API 文档页 | 可用 | 管理员用户与推荐码；API 说明展示 |

### 2. 技术与安全

- **后端**：Express（`apps/api/src/app.ts` 组装，`routes/index.ts` 挂载），领域在 `modules/*`；规则引擎 + 导出 + 认证 + 版本 + 团队存储。
- **前端**：统一为 **Next.js**（`ui/V0_SAAS`）；【历史说明，已下线】`apps/web`。
- **鉴权**：JWT；注册需推荐码；`admin` / `user`（业务上 user 与历史文档中的 operator 能力对齐）。
- **隔离**：导出历史、下载、会话等按 `ownerUserId`；前端 `localStorage` 按用户分 key。
- **版本**：服务端 `config/versions/records.json` 统一记录；删除子版本时校验是否被 `GL-` 总方案引用。

### 3. 文档体系

- V2 设计文档（架构 / API / 数据模型 / 权限）保留评审与历史口径。
- **实现与文档对齐说明** 单列于 `03_技术设计/系统演进/实现与文档对齐说明.md`。
- **预置模式** 产品口径补全于 `02_产品设计/规则与口径/PredefinedTemplate.md`。
- **环境变量** 见 `docs/ENVIRONMENT.md` 与根目录 `.env.example`。

### 4. 质量与测试

- 规则回归脚本与样本报告：`npm run rules:regression`、`rules:excel-report`；示例报告见 `05_测试与质量/测试报告/`。
- API / 规则单测以 `package.json` 中脚本为准；**带登录态的端到端自动化**仍较薄弱。

## 四、后续建议（优先级思路）

### R0（先行：需求分层与首发冻结）

1. **统一需求基线**：以 `01_需求管理/需求基线V1-首发与迭代分层清单.md` 作为唯一分层台账。
2. **冻结首发范围（严格 MVP）**：仅保留版本机制核心链路、四大评估主流程、鉴权与最小权限、评审最小闭环、发布可复现、契约一致。
3. **二轮/三轮边界固定**：二轮聚焦集成与数据层（含 WBS 增强），三轮聚焦产品完整闭环（评审/WBS/PLM）。
4. **数据策略约束**：当前不接 DB，维持单实例文件存储以验证流程与产品价值。

### P0（短期，稳定性与可交付）

1. **补齐部署文档与最小镜像**：`Dockerfile` + `docker-compose.yml` + `06_发布与部署` 下一页「首次部署检查清单」。
2. **同步 OpenAPI**：按 `实现与文档对齐说明` §3 补 paths，全局 `security: bearerAuth`，移除对 `X-Role` 的依赖描述。
3. **环境变量与密钥**：团队统一使用 `.env.example` + `docs/ENVIRONMENT.md`，生产密钥走密钥管理，不入库。

### P1（中期，产品闭环）

1. **标准PLM 预置**：定稿条目清单，更新 `presetModeConfigs` 与本文档 §3。
2. **WBS / 评审**：各定一页 MVP（只读关联总方案版本 + 占位图表 → 再迭代编辑能力）。
3. **E2E 测试**：选 Playwright/Cypress，覆盖登录 → 实施评估计算 → 导出 → 下载关键路径。

### P2（中长期，架构与扩展）

1. **后端持续演进**：在现有 `modules/*` 上收敛边界、补充观测与集成测试。
2. **前端策略**：以 `ui/V0_SAAS` 持续迭代，保持单前端体系。
3. **Agent 接口**：按 `项目开发TODO.md` 中 Agent 专项分阶段落地。
4. **企业集成**：云之家 / OIDC 单点登录单独立项（方案与评估已在历史对话中讨论）。
5. **数据库接入（第三轮）**：仅在迁移触发器满足时推进，且必须先更新计划与迁移评审文档。

### 首发统一门禁（与 README / TODO 对齐）

- `npm run build:web`
- `npm run build:api`
- `npm run test:modules`
- `npm run test:rules`
- `npm run test:integration`

### 首轮部署前工作计划（云部署简版，新增）

1. **选型与采购（Beta）**
   - 云主机建议：Ubuntu 22.04、`2C4G` 起步、`60G SSD`（建议 `80G` 更稳妥）。
   - 域名建议提前采购并完成备案准备。
2. **服务器初始化**
   - 创建普通运维用户，禁用密码登录，仅保留 SSH Key。
   - 仅开放 `22/80/443`，不直接暴露 `3000/3001`。
   - 安装 Docker / Docker Compose。
3. **项目上云与反向代理**
   - 准备生产 `.env`（`JWT_SECRET`、`KIMI_API_KEY` 等）。
   - `docker compose up --build` 启动 API + Web。
   - Nginx/Caddy 反代 `web:3001`，`/api` 反代 `api:3000`，配置 Let's Encrypt。
4. **数据持久化与运维**
   - `config/*` 挂载宿主机目录。
   - 每日定时备份（压缩，保留 7~14 天）。
   - 上线冒烟：`/api/v1/health` + 登录/版本流转/导出/团队协作主链路。
   - 建立发布脚本与故障 SOP（证书、磁盘、服务异常）。

### 采购前可先完成（不买云也可执行）

- 本地跑通：`docker compose up --build`
- 生成并评审生产环境变量清单
- 输出“上线检查清单 + 回滚步骤”
- 预选云厂商与实例规格，做到“采购即部署”

### Beta 数据治理三件事（新增）

- 每日自动备份 `config/*`（脚本：`npm run ops:backup:config`）。
- 启动或定时执行数据完整性校验（脚本：`npm run ops:check:config`）。
- 持续沉淀异常/修复日志（文件：`logs/data-anomaly-repair.log`）。

### 数据层迁移触发器（用于 DB 立项）

- 并发写冲突高频出现（建议：每周 >= 2 次人工修复）。
- 关键查询性能恶化（建议：P95 超过团队阈值）。
- 出现多实例部署/容器扩容刚需。
- 审计、统计、权限追溯需求明显增加。
