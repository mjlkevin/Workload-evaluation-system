01核心判断

一、当前系统的 3 个根本错位

  1. 把"PM"当一个人：实际上评估是 销售 → 实施顾问 → PM → PMO 的责任接力，当前系统只为"一个精算者"设计，没有接力语义。
  2. 把 AI 当格式转换器：Kimi 仅用于 Excel 解析、企业摘要两个离散点，没承担"需求分析师"的角色——抽取证据、查找缺口、生成问询。
  3. 把评估当"一个数"：真实交付包括 SOW、人天评估表、资源人天成本表、差异分析表、WBS、评估说明 Narrative 等至少 6 类结构化产物，当前系统只覆盖人天估算。

三、四条重构方向

1 · 角色分层交互　销售/售前走 Claude Skill + Agent（对话式），PM 和 PMO 保留 Web 工作台（结构化）。
2 · AI 从格式转换器升级为需求分析师　开放原始物料输入，做跨文档抽取 + 证据链 + 依赖推理 + 问询生成。
3 · 规则引擎一拆为二　「计算规则引擎」算人天，「完备性 / 依赖规则引擎」查缺漏，两者独立演化，均由 DSL 配置。
4 · 核心系统收敛为"可信源 + 稳定 API"　文件存储升级 Postgres，版本与审计保留，对三种前端（Skill / Agent / Web）统一提供服务。

02产品原则

以下 5 条原则是贯穿所有设计决策的根本约束，遇到取舍时回到这里。

P-1
AI Suggests, Human Decides
所有需承担责任的字段（人天 / 系数 / 风险 / 分期），AI 给行业参考 + 历史参考，人类保留最终决策权并承担责任。所有覆盖强制填写理由，理由沉淀为规则进化数据源。
P-2
责任接力
评估 ownership 在不同阶段传递（实施顾问 → PM → PMO）。接力是显式事件，保留全部前序轨迹，接手方可见原始思路与假设，决策权归当前 owner。
P-3
角色分层交互
销售/售前走对话式 Skill/Agent，PM/PMO 走结构化 Web。同一套核心 API、不同形态的前端——强行统一 UI 必然牺牲至少两个角色。
P-4
证据不入库，只留结论
原始文档（RFP / 微信 / 邮件 / 会议纪要）不作为文件存档，只沉淀抽取结论 + 段落摘录文本 + 来源标签。系统不是文档管理器，文档易改，结论稳定。
P-5
规则皆 DSL
计算规则和完备性规则全部用 DSL 描述，总部产品直接维护，不经开发发版。规则独立版本化 + 灰度 + 命中追溯。

接力事件的关键语义

  • 接力是显式事件：从 5a → 5c 不是"PM 自己打开看看"，是"实施顾问点'交接给 PM'"按钮，产生一条 AssessmentHandoff 记录。

  • 决策权显式传递：接手后实施顾问仍可 review 并提意见，但 PM 拥有修改权；冲突时以 PM 决策为准（见 TBD-5）。
  • 每次修正都区分语义：5b 的"交通费合计有误"是 Correction，5b 的"SOW 追加 → 加 20 人天"是 Revision，UI 和审计完全不同。

09DSL 与规则引擎

DSL = Domain-Specific Language（领域特定语言）

专为评估规则设计的简洁描述语法，让总部产品直接编辑、不经开发发版。

没有 DSL 的现状

规则埋在 TypeScript 里，改一条系数要走发版流程——周期 3-7 天。

有 DSL 之后

后台改 DSL → 历史样本回归测试 → 灰度到某组织 → 观察 → 全量——周期 30 分钟。

DSL 示例一 · 完备性规则

# 勾了生产，必须同时勾 BOM 主数据和库存基础档案
rule "生产模块基础数据完备性"
  when  勾选(生产)
  require  勾选(BOM主数据) and 勾选(库存基础档案)
  severity  high
  message  "勾了生产但缺 BOM 主数据源或库存基础档案"

DSL 示例二 · 计算规则

# 多组织推广：每多一个组织，实施人天 +15%
rule "多组织推广系数"
  when  组织数 > 1
  apply  实施人天 *= (1 + 0.15 * (组织数 - 1))

# 利民样本反推：HK 组织的 HKFRS 报表追加
rule "HK 国际会计准则报表"
  when  组织位置 == "HK" and 勾选(财务报表)
  apply  实施人天 += 20 * HK组织数
  explain  "HK 组织需按 BS & P&L 输出 HKFRS 报表，每个 +20 人天"

DSL 示例三 · 覆盖理由作为规则进化数据源

P-1 要求所有 AI 建议字段的人类覆盖强制填写理由。理由结构化后成为规则优化的反馈：

# 系统自动分析：过去 90 天，"医药难度系数" 从 0.2 被覆盖为 0.3 的次数 = 12
# 覆盖理由聚类：「GMP 验证复杂度高」(8) / 「批号追溯多层级」(4)
# → 管理后台自动提示：
#   规则 "医药行业难度系数" 建议从 0.2 调整为 0.3
#   支持证据：12 次 PM 覆盖，理由聚类 2 大类

11关键设计决策

D-1 ~ D-6 来自 v1，D-7 ~ D-15 为 v2 新增。

D-1销售一等公民 · 商机档案 v0 独立落库

销售快报阶段也应可追溯、可复盘；不独立沉淀等于放弃销售资产。
新增 OpportunityBrief；支持 parentProjectId 老客户继承（可选）。

D-2原始文档不入库 · 只存抽取结论 + 证据片段

文档频繁修改，不希望系统变成文档管理器。
Evidence 存 { 文档名, 段落文本, 抽取时间 }，不存原文；隐私压力显著下降。

D-3两套规则引擎 · 均用 DSL

规则频繁演化，总部产品需直接维护。
需要 DSL 语法 + 解释器；规则独立版本化 + 灰度 + 命中追溯。

D-4Skill 与 Web UI 不实时同步 · 变更以"重新提报"送达

谈判变更频繁，实时合并冲突成本高。
每次变更形成独立 ChangeSubmission；PM 侧看到有序列表 + 字段级 diff。

D-5初估交接包打置信度标签

让"要审多深"可量化。
4 维度合成：证据覆盖率 + 问询完整度 + 历史相似度 + 规则通过率；字段级 + 整包级两层。

D-6AI 默认 Kimi · ModelProvider 抽象

成本、合规、本地化部署可行性。
ModelProvider 接口覆盖长文抽取、JSON 输出、追问对话；Claude 作为可选实现。

D-7封版机制 · 成交后整体锁定 NEW

评估作为实施基线和合同依据，封版后不可随意变更。
AssessmentVersion.sealed 状态；封版后变更强制走"新评估版本"或"变更单"；封版触发点接销售的"发起合同流程"动作。

D-8评估协同工作区 · 沉淀群聊决策 NEW

真实工作流依赖拉群讨论（样本：14 人群），关键决策不能沉在群里丢失。
CollabWorkspace 承载问询-回复结构化；外部 IM 可并存，但决策回归系统。质询-回复自动沉淀为证据链。

D-9SOW 驱动评估 · 双向映射 NEW

SOW 是客户可见承诺，比评估更正式；SOW 变更必须驱动评估再审。
SOWDocument 条目与 AssessmentVersion 条目双向映射；SOW 变更 → 系统自动 diff → 告警"原评估未覆盖 X"。

D-10评估说明 Narrative 模板化 NEW

李加柱那段 289 人天说明是高度结构化的五段式，应该生成而非手打。
AssessmentNarrative 五段式模板：组织模块 / 数据治理 / 特殊场景 / 验收范围 / 周期成本；系统根据评估版本自动生成初稿，PM 编辑确认。

D-11开发评估独立 · 合同模式二选一 NEW

开发评估由独立开发顾问完成；合同形式（embedded / separate）决策受对外已报价约束。
DevAssessment 独立实体；contractMode 字段显式决策；私有云场景自动提示"评估部署运维"。

D-12Correction vs Revision 二元语义 NEW

样本中 "交通费合计有误" 是 Correction，"SOW 追加 → 加 20 人天" 是 Revision，两者审计路径不同。
AssessmentVersion.revisionType 字段；UI 区分两种事件；Correction 不改评估基线，Revision 产生新版本号。

D-13责任接力 · 显式事件 NEW

评估 ownership 在实施顾问 → PM 之间传递（P-2），不是单人持有。
AssessmentHandoff 事件实体；接手方看到全部前序轨迹；决策权随 ownership 传递。

D-14PMO 质量门 · 独立审核节点 NEW

PMO（魏春芳）是签约前最后一道关，审查交付物齐全性和方法论合规。
QualityGateReview 实体；PMO 工作台 UI；驳回理由按字段结构化；PMO 通过才能发起合同流程。

D-15交付模式分叉 · PublicCloud / PrivateCloud NEW

样本："本次按旗舰版公有云评估，私有云需额外拉开发评估部署运维"。
AssessmentVersion.deliveryMode 字段；PrivateCloud 时自动触发开发顾问的"部署运维评估"子任务。
