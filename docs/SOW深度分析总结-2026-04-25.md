# SOW（实施工作说明书）深度分析总结

> 基于三份文档的解析：
> - `XX项目_实施工作说明书(SOW)_2023简版.docx`（金额 < 10w 项目使用）
> - `XX项目_实施工作说明书(SOW)_2023标准版 .docx`（金额 ≥ 10w 项目使用）
> - `【利民集团】实施工作说明书(SOW)-V8.0-终稿.pdf`（真实项目实例）

---

## 一、简版 vs 标准版结构对比

| 章节 | 简版 | 标准版 | 说明 |
|---|---|---|---|
| 前言 | ✅ | ✅（+ 文档目的、相关文档） | 标准版更规范 |
| 项目综述（背景+目标） | ❌ | ✅ | 标准版才有，简版直接进入范围 |
| **项目范围** | ✅ | ✅ | **核心章节，两版都有** |
| — 产品范围 | ❌ | ✅ | 标准版明确列出主产品/第三方产品 |
| — **项目组织范围** | ✅ | ✅ | **检查项①** |
| — **实施业务范围** | ✅ | ✅ | **检查项②** |
| — **定制化开发范围** | ✅ | ✅ | **检查项③** |
| — 系统集成范围 | ✅ | ✅ | 标准版有详细接口说明 |
| — 初始数据转换范围 | ✅ | ✅ | 标准版有数据清单表格 |
| — 系统环境要求 | ❌ | ✅ | 标准版才有 |
| — 培训服务 | ❌ | ✅ | 标准版才有培训计划表格 |
| — 上线支持 | ✅ | ✅ | 简版一句话，标准版有支持清单 |
| 项目计划 | ✅（简化） | ✅（详细） | 标准版有实施策略、资源计划、WBS |
| — **WBS** | 附件 | ✅ 正文+附件 | **检查项④** |
| 项目组织及职责 | ✅（简化） | ✅（详细） | 标准版有角色职责矩阵 |
| 关键假设及约定 | ✅ | ✅（+ 项目暂停/终止条款） | 标准版更完善 |
| 项目验收 | ✅ | ✅（+ 验收标准/方法/内容） | 标准版有交付物签章清单 |
| 项目控制程序 | ❌ | ✅ | 标准版才有（变更/交付/争议） |
| 附件 | 5 个 | 5 个 | 定制化需求规格、WBS、变更申请单、任命书、盖章声明 |

### 核心差异一句话
> **简版 = 范围 + 计划 + 组织 + 验收（骨架）**  
> **标准版 = 简版 + 综述 + 产品范围 + 环境 + 培训 + 资源计划 + 控制程序（血肉）**

## 二、利民实例的关键发现

### 2.1 利民 SOW 完全遵循标准版模板

利民集团是真实项目（2026年1月签署），SOW 共 **40 页**，结构完全按标准版：

```
1 前言
2 项目综述（背景：数据孤岛、流程割裂；目标：集团化管控、成本精细化）
3 项目范围
  3.1 产品范围：金蝶AI星空套件公有云
  3.2 组织范围：3 家公司（香港、实业、南沙）
  3.3 业务范围：财务云+供应链云+制造云+人人业务云+开发服务云+多语言
  3.4 定制化开发范围：**无**
  3.5 系统集成范围：**无**
  3.6 初始数据转换：16 类主数据/期初数据
  3.7 系统环境：甲方负责硬件/网络/数据库
  3.8 培训：标准功能+数据收集+系统设置+关键用户+最终用户
  3.9 上线支持：3 个月（以月末结账为结束点）
4 项目计划
  4.1 实施策略：乙方主导推广，现场+远程
  4.2 总体计划：7 阶段（启动→需求→方案→构建→测试→上线→验收）
  4.3 资源计划：总计 460 人天（项目经理 109 + 财务顾问 117 + 供应链 117 + 制造 117）
  4.4 里程碑：蓝图确认、UAT确认、上线确认、验收确认
  4.5 WBS：详见附件
5 项目组织及职责
  5.1 组织架构
  5.2 甲方/乙方责任与义务（详细到办公设施清单：电话、网线、白板、投影仪...）
  5.3 角色与职责（项目领导小组、项目经理、关键用户、IT组、项目总监、实施顾问...）
6 关键假设及约定（20+ 条假设）
7 项目验收（1-4级问题定义、验收程序）
8 项目控制程序（PCR 变更流程、交付物接受、争议解决）
9-13 附件
```

### 2.2 实际项目与模板的偏差

| 偏差项 | 模板默认值 | 利民实例 | 说明 |
|---|---|---|---|
| 定制化开发 | "如存在..." | **无** | 实际项目可能没有定制开发 |
| 系统集成 | "如存在..." | **无** | 实际项目可能没有集成 |
| 模块范围 | 框架表格 | 已勾选具体模块 | 实际已明确 |
| 组织范围 | 框架表格 | 3 家具体公司 | 实际已明确 |
| 上线支持 | 框架 | 3 个月，以月末结账为结束点 | 有明确结束标准 |

### 2.3 对 T6 "SOW 完备性"规则的启示

**不能机械地要求"所有字段非空"**，因为：
1. 实际项目中"定制化开发范围"可能就是"无"
2. "系统集成范围"可能就是"无"
3. 这些"无"是**有效值**，不是缺失

**正确的校验逻辑应该是**：
- 这 4 个关键信息**必须有明确的证据**（即使证据值是"无"）
- 缺失 = 评估包中完全没有对应的证据字段

## 三、SOW 四大核心检查项（T6 规则字段边界）

基于三份文档 + TBD-V2-07 裁决，T6 `sow-completeness-v1` 规则应检查的 **4 个必填结构化字段**：

| # | 检查项 | 在 SOW 中对应章节 | 在评估包中可能的证据来源 | 允许"无"？ |
|---|---|---|---|---|
| 1 | **组织范围** | 3.2 项目组织范围 | `implementationScopeRows` / `basicInfo` | ❌ 必须有至少一家组织 |
| 2 | **模块范围** | 3.3 实施业务范围 | `productModuleRows` | ❌ 必须有至少一个模块 |
| 3 | **开发范围** | 3.4 定制化开发范围 | `devOverviewRows` | ✅ 可以是"无" |
| 4 | **WBS** | 4.5 工作分解结构 | `wbs`（或从评估包派生） | ❌ 必须有 WBS 结构 |

### 关于"其余不校验"

以下字段虽然在 SOW 中存在，但 T6 **不纳入自动校验**：
- 产品范围（通常由合同决定，不在评估阶段）
- 系统集成范围（可能为"无"，且涉及第三方接口，难以结构化）
- 初始数据转换范围（属于实施阶段细节，评估阶段无法确定）
- 系统环境要求（属于 IT 基础设施，评估阶段不涉及）
- 培训服务（属于实施计划，非范围核心）
- 上线支持（有标准模板，通常不自定义）

## 四、SOW 文档中的"可结构化" vs "不可结构化"内容

### 可结构化（系统可自动检查/生成）

| 内容 | 结构化方式 | T6 是否校验 |
|---|---|---|
| 组织范围清单 | 公司名 + 性质 | ✅ |
| 模块范围矩阵 | 模块 × 组织 × 期次 | ✅ |
| 开发功能清单 | 功能项 + 人天 | ✅ |
| WBS | 任务树（序号+名称+日期） | ✅ |
| 项目里程碑 | 名称 + 交付物 + 签章要求 | ❌（P1 再做） |
| 资源计划 | 角色 × 阶段 × 人天 | ❌（P1 再做） |

### 不可结构化（只能人工填写/审核）

| 内容 | 原因 |
|---|---|
| 项目背景/目标 | 自由文本，每家客户不同 |
| 实施策略说明 | 自由文本，需根据项目特点定制 |
| 双方职责详细描述 | 法律文本，有标准模板但需人工确认 |
| 关键假设条件 | 20+ 条标准假设，但可能需增删 |
| 验收标准细节 | 需根据客户要求调整 |
| 变更控制程序 | 标准流程，但金额阈值等需人工确认 |

## 五、对系统设计的直接影响

### 5.1 Evidence 字段路径设计

T6 规则消费的证据字段路径建议：

```
sow.organization_scope     ← implementationScopeRows 提取
sow.module_scope           ← productModuleRows 提取
sow.development_scope      ← devOverviewRows 提取（允许空数组）
sow.wbs                    ← 评估系统生成或人工维护
```

### 5.2 RuleViolation 产出示例

当某项目缺少组织范围证据时：

```ts
{
  ruleId: "sow-completeness-v1",
  severity: "error",      // 组织范围和模块范围是 error（必须有）
  fieldPath: "sow.organization_scope",
  message: "SOW 缺少项目组织范围：未找到实施主体组织清单",
  suggestion: "请在需求导入时提供 implementationScopeRows 或手动维护组织范围"
}
```

当开发范围明确为"无"时：

```ts
{
  ruleId: "sow-completeness-v1",
  severity: "info",       // 开发范围是 info（允许"无"）
  fieldPath: "sow.development_scope",
  message: "SOW 定制化开发范围已确认为'无'",
}
```

### 5.3 简版 vs 标准版的系统处理

系统应支持**两种 SOW 模式**：

| 模式 | 触发条件 | T6 校验强度 |
|---|---|---|
| 简版 | 合同金额 < 10w | 仅校验 4 项核心范围 |
| 标准版 | 合同金额 ≥ 10w | 校验 4 项核心 + 提示补充项目综述/控制程序 |

但 P0 阶段，T6 只做**一条规则**（不区分简版/标准版），仅校验 4 项核心范围是否齐全。

## 六、总结

> **SOW 是项目的"宪法"**，界定甲乙双方的权利义务边界。三份文档揭示了一个清晰的层次结构：
>
> 1. **核心骨架**（简版就有）：范围 + 计划 + 组织 + 验收
> 2. **完整血肉**（标准版才有）：综述 + 产品 + 环境 + 培训 + 资源 + 控制程序
> 3. **实际填充**（利民实例）：在模板骨架上填入具体项目信息
>
> **T6 的 `sow-completeness-v1` 规则只需守住骨架的 4 根支柱**：
> - 组织范围（有没有实施主体？）
> - 模块范围（上哪些功能？）
> - 开发范围（有没有定制？允许"无"）
> - WBS（工作怎么分解？）
>
> 其余内容属于"血肉"，在评估阶段无法结构化校验，留给人工审核和 P1 的 PMO 检查清单处理。
