## 1. 背景与动机

2026 年 03-04 月初版 `apps/api/src/services/ai.service.ts` 膨胀到 **2541 行**，承载了 Kimi API 调用、Excel 解析、企业画像抽取、规则回退、评估草稿生成等全部 AI 相关逻辑。直接问题：

- **单文件难以测试**：业务规则与 HTTP 细节混在一起，无法独立 mock。
- **AI 厂商锁定**：Kimi endpoint / payload 硬编码在各业务函数里，换 provider 需要动几十处。
- **证据链断裂**：AI 输出直接进入 ExcelRow，人工修改无法追溯；规则降级理由散落在 try/catch 文案里。
- **与 v2 设计不兼容**：v2 提出"7 角色责任接力"、"DSL 完备性检查"、"证据驱动的评审"，这些能力无法在现有单体结构上叠加。

P0 的目标：**搭建能承接 v2 能力的基座**，不破坏现有功能。

## 3. P0 范围边界

### ✅ 做

1. **AI 基座抽象层**：`ModelProvider` 接口 + `KimiProvider` 实现 + 注册表。
2. **证据链契约**：`Evidence` / `ExtractionResult` 类型 + JSON Schema。
3. **长文抽取器**：从 Excel / 文本产出 `ExtractionResult`。
4. **渐进迁移**：旧 `ai.service.ts` 调用点改走新 Provider，但保留外部接口签名。
5. **持久化升级**：引入 Postgres + Drizzle ORM，`evidences` / `extraction_results` / `change_logs` 进 PG。
6. **DSL 引擎雏形**：实现一条"SOW 完备性"规则作为概念验证。
7. **7 角色 RBAC**：角色常量 + 权限矩阵（仅后端约束，不动前端 UI）。
8. **冒烟测试**：IA 初评一条 happy path 走通 Vitest。

### ❌ 不做（推迟到 P1+）

- 不动前端 UI（V0_SAAS）。
- 不迁旧 JSON stores（users / versions / templates / system configs）。
- 不接入第二个 AI provider（OpenAI / 通义 etc.）。
- 不引入 BullMQ / 任务队列。
- 不实现完整 DSL 规则集（只做 1 条）。

## 5. 关键设计决策台账

> 原则：**每项决策必须有理由、有时间；有替代方案者写明为何不选**。

| ID | 日期 | 决策 | 理由 | 替代方案 |
|---|---|---|---|---|
| D-01 | 2026-04-24 | **Provider 接口内置重试/超时，非调用方责任** | 与旧 `requestKimiCompletion` 一致，避免每个业务点重复写重试 | 暴露裸 `fetch` 让调用方自行处理（放弃：违反"调用方只管业务"原则） |
| D-02 | 2026-04-24 | **ProviderError 保留 `legacyReason` 字段** | `toFriendlyFallbackReason` 旧文案还在用，改造需渐进 | 直接断掉兼容性（放弃：T4 一次性迁移风险过大） |
| D-03 | 2026-04-24 | **Evidence.history 追加而非替换** | v2 强调"不可抵赖"，人工修改不能抹除 AI 原始证据 | 用 change_logs 表记录（仍保留，但 history 字段提供快速可见性） |
| D-04 | 2026-04-24 | **confidence 三档而非自由浮点** | 规则=1.0、人工=1.0、AI=模型自报；简化 DSL 阈值决策 | AI 用 logprobs 计算置信度（放弃：Kimi 未必稳定暴露） |
| D-05 | 2026-04-24 | **引入 Postgres**（推翻最初"P0 继续用 JSON"约束） | 证据链的 relational join 在 JSON 下几乎无法做；本机已装 Postgres.app 无额外部署成本 | 继续 JSON（放弃：T5 change_log 查询能力不足） |
| D-06 | 2026-04-24 | **Drizzle ORM**，非 Prisma / TypeORM / raw `pg` | TS-first、零 codegen 运行时、SQL 可见、CJS 兼容好 | Prisma（放弃：CJS 有坑、codegen 重）；raw `pg`（放弃：repository 维护成本高） |
| D-07 | 2026-04-24 | **渐进迁移**（P0 只新增表，旧 JSON 不动） | 一次性全迁阻塞 T5、风险大；users/versions 的 schema 需要单独设计 | 全量迁移（放弃：工作量翻倍） |
| D-08 | 2026-04-24 | **Schema 约定**：uuid 主键、snake_case、timestamptz、jsonb、硬删 | PG 主流实践；uuid 与现有 `randomUUID()` 直接对应；jsonb 容纳联合类型 | bigserial / camelCase / 软删（放弃：见下方注） |
| D-09 | 2026-04-24 | **迁移手动触发**，不在应用启动时自动跑 | 生产部署需要显式审批；开发机也便于回滚 | 启动自动 migrate（放弃：一次手滑全库变更） |
| D-10 | 2026-04-24 | **测试策略**：独立 `workload_eval_test` + 事务回滚 | 事务回滚比 TRUNCATE 快；独立库避免污染 dev 数据 | testcontainers（推迟到 CI） |

**注（D-08）**：
- 不用 bigserial：uuid 可在应用层生成，不依赖 DB 单调序号，未来可分库。
- 不用 camelCase：PG 默认会把 camelCase 转小写，除非加双引号；统一 snake_case 省心。
- 不用 deleted_at 软删：P0 简化；审计由 change_logs 表承担。

## 8. 本地环境与命令速查

### 环境

- **PG**：Postgres.app 18.3，监听 `localhost:5432`
- **两个库**：`workload_eval`（dev）、`workload_eval_test`（vitest）
- **owner**：当前系统用户（本机 `kevin`，部署到他机时要调整）
- **连接串**：`postgres://kevin@localhost:5432/workload_eval`（写在 `apps/api/.env.local`，不 commit）

### 常用命令

```bash
# 开发启动（和以前一样）
npm run dev:api                                          # 后端
npm run dev:web                                          # 前端

# DB 迁移工作流
npm run db:generate -w apps/api                          # 改 schema 后，生成 SQL migration
npm run db:migrate   -w apps/api                         # 应用到开发库 workload_eval
DATABASE_URL=postgres://kevin@localhost:5432/workload_eval_test \
  npm run db:migrate -w apps/api                         # 应用到测试库
npm run db:studio    -w apps/api                         # 可视化表 & 数据（drizzle-kit）

# 测试
npm run test:modules  -w apps/api                        # 既有单元/用例/handler
npm run test:rules    -w apps/api
npm run test:integration -w apps/api

# 类型检查
cd apps/api && npx tsc --noEmit                          # 三个 exports.module 报错是历史遗留，忽略
```

### 新机器冷启动清单

1. 装 PG 18.x（Postgres.app 或 brew）
2. `createdb workload_eval && createdb workload_eval_test`
3. `cp apps/api/.env.example apps/api/.env.local`，填 `DATABASE_URL`
4. `npm install`
5. 对两库各跑一次 `npm run db:migrate`

## 附录 B：PG Schema 演进规则

> 未来 T5 及之后所有改 schema 的流程

1. 改 `apps/api/src/db/schema/*.ts`
2. `npm run db:generate -w apps/api` → 生成 `drizzle/NNNN_xxx.sql`
3. **肉眼 review 生成的 SQL**（drizzle-kit 不完美，偶有 DROP/CREATE 歧义）
4. `npm run db:migrate -w apps/api` 应用到开发库
5. `DATABASE_URL=... npm run db:migrate` 应用到测试库
6. 生成的 SQL + meta 文件 **必须** commit，和代码变更放同一个 PR
7. 不允许手动修改已合并的历史 migration；需要改就新增一条 migration 补丁
