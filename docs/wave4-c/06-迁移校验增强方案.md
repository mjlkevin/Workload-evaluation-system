# Wave4-C2 · 迁移校验增强方案

## 结论速览

迁移底座已从"能跑"升级为"可验证、可追踪、可回放"。当前实现了 14 项深度校验，覆盖数量、唯一性、完整性、payload 兼容性、重复数据识别。全部 14 项中 13 项通过、1 项警告（源内测试数据重复，非阻塞）。

## 2. 校验方式

### 2.1 基础校验（--validate）
仅做数量对账，适合快速确认迁移是否完整。

```bash
npm run migration:v1-to-v2 -- --validate
```

### 2.2 深度校验（--comprehensive-validate）
执行全部 14 项校验，输出分类结果。

```bash
npm run migration:v1-to-v2 -- --comprehensive-validate
```

### 2.3 迁移后自动校验
正常迁移模式下，基础校验和深度校验会自动在迁移后执行。

## 4. 失败处理

| 失败场景 | 处理方式 | 是否阻塞 |
|---------|---------|---------|
| 数量不对账 | 控制台 ❌ + 报告标记 fail + 进程 exit 1 | 是 |
| 唯一性冲突 | 控制台 ❌ + 报告列出重复项 + 进程 exit 1 | 是 |
| 必填字段为空 | 控制台 ❌ + 报告列出 null 统计 + 进程 exit 1 | 是 |
| status 枚举非法 | 控制台 ❌ + 报告列出非法记录 + 进程 exit 1 | 是 |
| 源内重复数据 | 控制台 ⚠️ + 报告标记 warn，进程 exit 0 | 否 |
| payload 兼容字段缺失 | 控制台 ⚠️ + 报告标记 warn，进程 exit 0 | 否 |

## 5. 重跑策略

迁移脚本本身幂等：
- **users**：按 `username` 跳过已存在
- **version_code_rules**：按 `ruleId` 跳过已存在
- **records**：按 `versionCode` 跳过已存在 + 源内去重

因此重跑策略为：
1. 修复源数据或 PG 数据问题
2. 直接重跑 `npm run migration:v1-to-v2`
3. 已迁移的记录自动跳过，新增/修复的记录自动插入
4. 迁移后自动执行深度校验确认结果

## 6. 异常记录输出

### 6.1 被跳过的记录
迁移日志会明确输出跳过原因：
- `[migrate:records] 跳过已存在: RI-20260328001A` — 数据库已有
- `[migrate:records] 跳过源内重复 versionCode: GL-UT-03 (id=ver-ut-gl-1)` — 源内重复

### 6.2 被去重的记录
深度校验报告中 DUPLICATE 分类会列出：
- `versionCode` 重复值
- 重复次数

### 6.3 无法转换的记录
迁移日志会输出：
- `[migrate:records] 插入失败: <versionCode> — <error_message>`
- 常见原因：非 UUID id（已自动处理）、唯一约束冲突、字段类型不匹配

### 6.4 需要人工确认的记录
以下情况会标记为 warn 但不阻塞：
- 源内重复（需确认保留哪一条）
- payload 兼容字段缺失（需确认前端是否受影响）
