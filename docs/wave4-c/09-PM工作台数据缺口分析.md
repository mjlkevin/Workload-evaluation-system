# Wave4-C2 · PM 工作台数据缺口分析

## 结论速览

PM 工作台的 5 个核心语义区（接力区、版本区、交付物区、审核区、封版区）中，**个体服务都已就绪，但缺少统一的聚合视图**。最大的缺口是：
1. **无复合查询接口** — 前端需要发 4-5 个请求才能拼出一个版本的全貌
2. **无 FK 与 Relations** — 无法做关系查询，所有关联都是软引用
3. **Deliverable 内容不可编辑** — 生成后只能改状态，不能改内容
4. **HistoryProject 缺少 `projectName`** — 列表视图无法直接展示可读名称

---

## 1. 接力区（AssessmentHandoff）

### 当前能力
- Schema：`assessment_handoffs` 表完整
- Service：`create`, `findById`, `listByVersion`, `listByToRole`, `update`, `delete`
- Routes：`POST/GET/PATCH /handoffs`

### 数据缺口

| 缺口 | 影响面 | 阻塞程度 | 说明 |
|-----|--------|---------|------|
| 缺少 `GET /versions/:versionId/handoffs` | 版本详情页展示接力历史 | ⚠️ 中 | Service 有 `listByVersion`，但无路由 |
| 无 FK → assessment_versions | 数据一致性 | ⚠️ 中 | 软引用，删除版本时 handoff 残留 |
| 无 `initiatedAt` / `acceptedAt` 独立字段 | 时间线精确展示 | 🟡 低 | 可用 `createdAt` / `updatedAt` 近似 |
| 无 `rejectedReason` 字段 | 拒绝原因记录 | 🟡 低 | 目前用通用 `notes` 字段代替 |

### 是否阻塞
- **PM 工作台 inbox**：❌ 不阻塞，`listByToRole` 已可用
- **版本详情页接力历史**：⚠️ 需要补路由

---

## 2. 版本区（AssessmentVersion）

### 当前能力
- Schema：`assessment_versions` 表完整，含状态机、payload、ownerRole
- 迁移后已有 23 条历史记录

### 数据缺口

| 缺口 | 影响面 | 阻塞程度 | 说明 |
|-----|--------|---------|------|
| 无 `projectId` / `opportunityId` | 项目维度聚合 | 🔴 高 | 无法直接查询"某项目的所有版本" |
| `payload` 无类型约束 | 前端解析风险 | 🟡 中 | 不同 type 的 payload 结构不同 |
| `linkedNarrativeId` 从未更新 | 叙事关联失效 | 🟡 中 | narrative 创建后不会回写 version |
| 缺少统一聚合接口 | PM 工作台总览 | 🔴 高 | 前端需分别查 version + handoff + deliverable + review + seal |

### 是否阻塞
- **版本历史列表**：❌ 不阻塞，已有 `listByVersion` 语义
- **PM 工作台总览**：🔴 阻塞，需复合接口或前端 orchestration

---

## 3. 交付物区（Deliverable）

### 当前能力
- Schema：`deliverables` 表完整，含 4 种类型
- Service：`generateAll`, `findById`, `listByVersion`, `listByVersionAndType`, `updateStatus`, `delete`
- Routes：已挂载

### 数据缺口

| 缺口 | 影响面 | 阻塞程度 | 说明 |
|-----|--------|---------|------|
| 无 `updateContent` 方法 | PM 手动编辑交付物 | 🔴 高 | 生成后只能确认/取消，不能改内容 |
| `variance_analysis` 为 noop | 差异分析展示 | 🟡 中 | 始终返回 0% 差异，因为 baseline = current |
| `ratePerDay` 硬编码 3000 | 成本计算准确性 | 🟡 中 | 注释说明未来接入 RateCard |
| 无 `confirmedByUserId` / `confirmedAt` | 审计追踪 | 🟡 低 | 缺少确认人记录 |
| 无导出 URL / file 字段 | 下载交付物 | 🟡 中 | 目前只有 JSON content |

### 是否阻塞
- **交付物列表 + 确认**：❌ 不阻塞
- **PM 手动调整交付物内容**：🔴 阻塞
- **差异分析展示**：🟡 有功能但数据无意义

---

## 4. 审核区（QualityGateReview）

### 当前能力
- Schema：`quality_gate_reviews` 表完整，含 checklist、verdict、rejectionReasons
- Service：`create`, `autoReview`, `findById`, `findByVersionId`, `update`, `delete`
- `autoReview` 已实现自动化审核逻辑

### 数据缺口

| 缺口 | 影响面 | 阻塞程度 | 说明 |
|-----|--------|---------|------|
| `methodologySevenPhases` 硬编码 true | 审核准确性 | 🟡 中 | autoReview 中永远通过 |
| `rateCardCorrect` 硬编码 true | 审核准确性 | 🟡 中 | autoReview 中永远通过 |
| 无 `reviewRound` 字段 | 多轮审核记录 | 🟡 低 | 只能靠 `createdAt` 区分 |
| 无审核 → seal 的自动联动 | 工作流自动化 | 🟡 低 | 审核通过不会自动触发 seal |

### 是否阻塞
- **手动审核**：❌ 不阻塞
- **自动审核**：⚠️ 部分通过（2/5 checklist 项硬编码为 true）

---

## 5. 封版区（SealedBaseline）

### 当前能力
- Schema：`sealed_baselines` 表完整，含 artifactsSnapshot、contractFlowId
- Service：`seal`, `findById`, `findByVersionId`, `listByStatus`, `supersede`, `delete`

### 数据缺口

| 缺口 | 影响面 | 阻塞程度 | 说明 |
|-----|--------|---------|------|
| `artifactsSnapshot` 需调用方构造 | seal 操作易错 | 🟡 中 | 不会自动收集当前 deliverables |
| 无 `unseal` / `reopen` 方法 | 封版后回退 | 🟡 低 | 只能通过 `supersede` 创建新版 |
| 无 `sealedAt` 独立字段 | 精确时间线 | 🟡 低 | 可用 `createdAt` 近似 |
| 无 webhook / 事件发射 | 下游合同系统联动 | 🟡 低 | 注释提到但未实现 |

### 是否阻塞
- **封版操作**：❌ 不阻塞
- **自动快照**：⚠️ 需要前端/service 层手动构建 artifactsSnapshot

## 6. 叙事区（AssessmentNarrative）

### 当前能力
- Schema：`assessment_narratives` 表完整，含 5 段落 + metadata
- Service：`create`, `generateDraft`, `findById`, `findByVersionId`, `update`, `delete`
- `generateDraft` 已实现从 packData + estimateData 自动生成

### 数据缺口

| 缺口 | 影响面 | 阻塞程度 | 说明 |
|-----|--------|---------|------|
| `linkedNarrativeId` 未同步 | version → narrative 关联 | 🟡 中 | narrative 创建后不回写 assessment_versions |
| 无 diff/history 追踪 | 叙事变更审计 | 🟡 低 | 只有 `metadata` JSON 占位 |
| `templateId` 未实际使用 | 模板化生成 | 🟡 低 | generateDraft 硬编码 `generatedFrom: "template"` |

### 是否阻塞
- **叙事生成与编辑**：❌ 不阻塞

## 8. 最优先补齐建议

1. **🔴 P0：补 `GET /versions/:versionId/workbench` 复合接口**
   - 聚合 version + handoffs + deliverables + review + seal + narrative
   - 这是 PM 工作台总览页面的数据底座

2. **🔴 P0：补 Deliverable `updateContent` 能力**
   - 允许 PM 在生成后手动调整交付物内容

3. **⚠️ P1：补 `GET /versions/:versionId/handoffs` 路由**
   - 版本详情页展示接力历史

4. **⚠️ P1：history_projects 补 `projectName` 字段**
   - 历史项目列表需要可读名称

5. **🟡 P2：补 FK + Drizzle Relations**
   - 长期数据一致性保障，但非前端阻塞
