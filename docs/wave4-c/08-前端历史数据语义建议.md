# Wave4-C2 · 前端历史数据语义建议

## 结论速览

迁移后的 `assessment_versions` 已包含前端所需的全部兼容字段（`projectName`、`checkoutStatus`、`versionDocStatus`、`isHistoricalArchive` 等）。前端可以开始消费 PG 数据，但建议通过 Service 层封装后再使用，避免直接绑定 `payload` 内部结构。`users` 和 `version_code_rules` 表字段稳定，可直接使用。

---

## 1. assessment_versions（核心历史数据表）

### 1.1 前端使用场景

| 场景 | 消费字段 | 说明 |
|-----|---------|------|
| 版本历史列表 | `versionCode`, `status`, `updatedAt`, `payload.projectName` | 展示版本号、状态、更新时间、项目名称 |
| VCS 工具栏 | `checkoutStatus`, `versionDocStatus`, `checkedOutByUserId` | 控制检出/检入按钮状态 |
| 历史归档标记 | `payload.isHistoricalArchive` | 标识是否为历史归档版本 |
| 状态流转 | `status` (draft → checked_out → checked_in → promoted → sealed) | 驱动工作流状态机 |
| 责任角色 | `ownerRole` (IMPL / PM / PMO / ADMIN) | 显示当前责任方 |

### 1.2 字段稳定性分级

| 字段 | 稳定性 | 说明 |
|-----|--------|------|
| `assessmentVersionId` | 🟢 稳定 | UUID 主键，永不变 |
| `versionCode` | 🟢 稳定 | 业务版本号，唯一 |
| `status` | 🟢 稳定 | v2 状态机，枚举固定 |
| `ownerRole` | 🟢 稳定 | v2 角色枚举 |
| `revisionType` | 🟢 稳定 | initial / correction / revision |
| `createdAt` / `updatedAt` | 🟢 稳定 | 标准时间戳 |
| `payload.projectName` | 🟡 可选 | 从 v1 payload 提取，可能为空字符串 |
| `payload.checkoutStatus` | 🟡 可选 | v1 兼容字段，默认 "checked_in" |
| `payload.versionDocStatus` | 🟡 可选 | v1 兼容字段，默认 "draft" |
| `payload.isHistoricalArchive` | 🟡 可选 | v1 兼容字段，默认 false |
| `payload._v1Type` | 🔴 内部 | 仅用于回查，不建议前端绑定 |
| `payload._v1*` | 🔴 内部 | 全部不建议前端依赖 |
| `payload` 其他字段 | 🟡 可选 | 业务数据，结构随类型变化 |

### 1.3 不建议前端直接绑定的字段

| 字段 | 原因 |
|-----|------|
| `payload._v1Type` | 内部保留字段，未来可能移除 |
| `payload._v1TemplateId` | 内部保留字段 |
| `payload._v1CreatedByUserId` | 内部保留字段 |
| `payload._v1MajorLetter` / `_v1MinorNumber` | 内部保留字段 |
| `linkedSowId` / `linkedNarrativeId` | 软引用，可能为 null，且未来会补 FK |
| `parentVersionId` | 仅用于版本血缘，一般不在列表展示 |

## 2. users 表

### 2.1 前端使用场景

| 场景 | 消费字段 |
|-----|---------|
| 用户列表 / 管理 | `userId`, `username`, `role`, `status`, `createdAt`, `lastLoginAt` |
| 权限控制 | `role` (admin / sub_admin / user) |
| 检出者显示 | `username`（通过 `checkedOutByUserId` 关联） |

### 2.2 字段稳定性

全部字段为 🟢 稳定。`role` 枚举当前为 `admin / sub_admin / user`，v2 RBAC 通过 `legacyRoleToV2Roles` 映射到 `ADMIN / SALES / PRE_SALES / IMPL / PM / DEV / PMO`。

---

## 3. version_code_rules 表

### 3.1 前端使用场景

| 场景 | 消费字段 |
|-----|---------|
| 版本号生成规则展示 | `ruleId`, `moduleKey`, `moduleName`, `moduleCode`, `prefix`, `format`, `sample` |
| 规则状态管理 | `status`, `effectiveAt` |

### 3.2 字段稳定性

全部字段为 🟢 稳定。该表为配置表，行数少（当前 6 条），适合前端缓存。

---

## 4. 前端消费建议

### 4.1 建议通过 Service 层封装

不要直接在前端组件里写：
```tsx
// ❌ 不推荐：直接读取 payload 内部结构
const projectName = version.payload?.basicInfo?.projectName || version.payload?.projectName;
```

建议由 Service 层提供：
```tsx
// ✅ 推荐：通过 Service 获取结构化数据
const { projectName, checkoutStatus, versionDocStatus } = await workloadService.getVersionDisplayMeta(versionId);
```

### 4.2 建议封装的新接口

| 接口 | 用途 | 返回字段 |
|-----|------|---------|
| `GET /versions/:id/display-meta` | 获取版本展示元数据 | `projectName`, `statusLabel`, `checkoutStatusLabel`, `isHistoricalArchive`, `ownerRoleLabel` |
| `GET /versions/:id/history` | 获取版本历史时间线 | 按 `updatedAt` 排序的记录列表 |
| `GET /versions/grouped` | 按项目分组版本 | `{ projectName, globalVersion, modules: [...] }` |

### 4.3 风险字段处理

| 风险 | 建议 |
|-----|------|
| `payload` 结构不统一（不同 type 有不同结构） | Service 层按 type 做分支解析，前端只消费统一后的字段 |
| `payload.projectName` 可能为空 | 前端展示时提供兜底文案（如"未命名项目"） |
| v1 状态值与 v2 枚举不完全对齐 | 通过 Service 层的 `mapStatusToLabel` 统一转换 |
