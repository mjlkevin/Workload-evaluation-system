# Wave4-C · 前端接入建议

> 目标：迁移完成后，前端工作台能够真正继承历史上下文
> 基线：7c00005

## 一、前端当前数据消费模式

### 1.1 版本历史弹窗（version-history-dialog.tsx）

**消费的数据**：
- `records` 列表（来自 `/api/v1/versions`）
- 每条记录的：`versionCode`、`updatedAt`、`checkoutStatus`、`versionDocStatus`、`status`、`isHistoricalArchive`、`payload`
- 从 `payload` 中提取：`projectName`、`checkinNote`

**迁移后变化**：
- 数据来源从 JSON store 变为 `assessment_versions` PG 表
- `payload` 字段仍然保留完整 JSON 结构，前端读取方式**不变**
- 但 `payload` 中新增了 `_v1Meta` 字段（迁移标识），前端应**忽略**该字段

**建议**：
```typescript
// 前端读取 payload 时，过滤掉 _v1Meta
const cleanPayload = (payload: Record<string, unknown>) => {
  const { _v1Meta, ...rest } = payload;
  return rest;
};
```

### 1.2 版本 VCS 工具栏（version-vcs-toolbar.tsx）

**消费的数据**：
- checkout/checkin/promote 等操作（来自 `/api/v1/versions/:id/checkout` 等）
- `checkoutStatus`、`versionDocStatus`

**迁移后变化**：
- v1 `status` 已映射为 v2 `status` + `checked_out_by_user_id`
- `checkout` 操作返回的 `checkoutStatus` 仍为 `"checked_out"`，前端无需修改
- `promote` 操作后 v2 status 变为 `"promoted"`，前端需确认展示文案

### 1.3 工作台 Service 层（workload-service.ts）

**消费的数据**：
- `listModuleVersions()` → `/api/v1/versions?type=xxx`
- `listGlobalVersions()` → `/api/v1/versions?type=global`
- `listTemplateSummaries()` → `/api/v1/templates`
- `listTeamReviews()` → `/api/v1/teams/:id/reviews`
- `listUsers()` → `/api/v1/auth/users`

**迁移后变化**：

| API | 当前数据源 | 迁移后数据源 | 前端需修改 |
|---|---|---|---|
| `/api/v1/versions/*` | JSON store | PG `assessment_versions` | ❌ 无需修改 |
| `/api/v1/templates/*` | JSON store | PG `templates` + `template_items` | ⚠️ 需确认字段 |
| `/api/v1/teams/*` | JSON store | PG `teams` + `team_members` | ⚠️ 需确认字段 |
| `/api/v1/auth/users` | JSON store | PG `users` | ❌ 无需修改 |
| `/api/v1/system/version-code-rules` | JSON store | PG `version_code_rules` | ❌ 无需修改 |

## 二、哪些历史数据会被前端直接消费

### P0：必须优先迁移（前端强依赖）

| 数据 | 前端使用场景 | 迁移后 API |
|---|---|---|
| `assessment_versions` | 版本历史弹窗、VCS 工具栏、工作台 | 已有 `/api/v1/versions/*` |
| `users` | 用户管理、登录态、权限展示 | 已有 `/api/v1/auth/*` |
| `version_code_rules` | 版本创建时编码规则 | 已有 `/api/v1/system/version-code-rules` |

### P1：建议迁移（前端中度依赖）

| 数据 | 前端使用场景 | 迁移后 API |
|---|---|---|
| `templates` | 模板选择页、导入页 | 已有 `/api/v1/templates/*` |
| `teams` | 团队管理、评审、评论 | 已有 `/api/v1/teams/*` |

### P2：可延后（前端弱依赖或后台管理）

| 数据 | 前端使用场景 | 迁移后 API |
|---|---|---|
| `invite_codes` | 后台邀请码管理 | 已有 `/api/v1/auth/invite-codes` |
| `implementation-dependency-rules` | 规则引擎后台 | 已有 `/api/v1/system/implementation-dependency-rules` |
| `requirement-settings` | 系统配置后台 | 已有 `/api/v1/system/requirement-settings` |

## 三、哪些数据是「历史基线」

**定义**：历史基线 = 一旦写入就不应再修改、用于回溯和参考的数据。

| 数据 | 是否历史基线 | 说明 |
|---|---|---|
| `assessment_versions`（status=sealed） | ✅ 是 | 封版后锁定，不可修改 |
| `assessment_versions`（status=draft） | ❌ 否 | 草稿可继续编辑 |
| `history_projects` | ✅ 是 | 结案后沉淀，只读参考 |
| `sealed_baselines` | ✅ 是 | 封版基线，只读 |
| `templates` | ❌ 否 | 模板可更新 |
| `users` | ❌ 否 | 用户状态可变更 |

## 五、前端后续建议

### 5.1 短期（Wave 4 内）

1. **保持 payload 读取方式不变**：v1 records 的 payload 整体进入 v2 `assessment_versions.payload`，前端无需修改字段路径
2. **过滤 _v1Meta**：在 service 层统一过滤掉 payload 中的 `_v1Meta` 字段
3. **验证 status 展示文案**：确认 `promoted`、`sealed` 等 v2 status 在前端的展示文案是否准确

### 5.2 中期（Wave 5）

1. **模板数据 PG 化后**：`getTemplateDetail()` 需改为 JOIN `templates` + `template_items`
2. **团队数据 PG 化后**：`getTeamDetail()` 需改为 JOIN `teams` + `team_members`
3. **历史项目推荐**：在评估页接入 `history_projects.findSimilar()` API

### 5.3 长期（Wave 6+）

1. **逐步清理 payload 冗余**：将 payload 中的常用字段（如 `projectName`、`customerName`）提升到 PG 独立字段，便于索引和查询
2. **版本历史搜索**：基于 PG 的 `projectName` 字段实现版本历史搜索

## 六、API 契约建议

为保证前端平滑过渡，建议后端在迁移后保持以下 API 契约：

```typescript
// 版本记录 DTO 保持不变
interface VersionRecordDto {
  id: string;
  type: VersionType;
  versionCode: string;
  status: string;
  payload: Record<string, unknown>;
  createdAt: string;
  updatedAt: string;
  createdByUserId?: string;
  createdByUsername?: string;
  updatedByUserId?: string;
  updatedByUsername?: string;
  checkoutStatus?: "checked_in" | "checked_out";
  versionDocStatus?: "drafting" | "reviewed";
  checkedOutByUserId?: string;
  checkedOutByUsername?: string;
  checkoutAt?: string;
  majorLetter?: string;
  minorNumber?: number;
  baseCode?: string;
  isHistoricalArchive?: boolean;
  archivedAt?: string;
}
```

**关键**：即使后端数据源从 JSON 变为 PG，API 返回的字段名和类型必须保持一致。
