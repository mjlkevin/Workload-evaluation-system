# Wave4-C2 · 前端接口说明

## 结论速览

前端可直接消费的接口已覆盖 80% 场景。当前最缺的是：
1. **统一版本工作台接口** — 一个请求拿全 version + handoff + deliverable + review + seal
2. **历史项目摘要接口** — 不同角色看不同字段
3. **版本分组列表接口** — 后端按 `globalVersionCode` 分组，替代前端 60 行 client-side 聚合

---

## 1. 可直接消费的接口（🟢 稳定）

### 1.1 认证与用户

| 接口 | 方法 | 返回 | 稳定性 |
|-----|------|------|--------|
| `POST /auth/login` | POST | JWT token | 🟢 |
| `GET /auth/users` | GET | 用户列表 | 🟢 |
| `GET /auth/me` | GET | 当前用户 | 🟢 |

### 1.2 历史项目库

| 接口 | 方法 | 返回 | 稳定性 |
|-----|------|------|--------|
| `GET /history/projects` | GET | 历史项目列表（支持 industry/scale 过滤） | 🟢 |
| `GET /history/projects/:id` | GET | 历史项目详情 | 🟢 |
| `GET /history/similar?industry=...&scale=...` | GET | 相似项目推荐 | 🟢 |
| `POST /history/projects` | POST | 创建历史项目 | 🟢 |
| `PATCH /history/projects/:id` | PATCH | 更新历史项目 | 🟢 |

> ⚠️ 注意：`GET /history/projects` 返回列表无 `projectName`（字段缺失），当前需用 `industry + scale` 拼接展示名。

### 1.3 PM 工作台 — 接力

### 1.4 PM 工作台 — 叙事

| `PATCH /pm/narratives/:id` | PATCH | 更新叙事段落 | 🟢 |

### 1.5 PM 工作台 — 交付物

| 接口 | 方法 | 返回 | 稳定性 |

## 2. 需要 Service 层封装的接口（🟡 建议封装）

### 2.1 版本分组列表

**当前前端做法：**
```tsx
// 前端发 5 个请求，client-side 按 globalVersionCode 分组
const [global, assessment, resource, requirement, dev] = await Promise.all([
  listVersions('global'),
  listVersions('assessment'),
  // ...
]);
// 60 行分组逻辑
```

**建议后端提供：**
```
GET /versions/grouped?projectName=...
```

```json
{
  "groups": [
    {
      "globalVersionCode": "GL04001",
      "projectName": "蓝海集团-星空AI",
      "latestGlobal": { /* ... */ },
      "modules": {
        "requirementImport": { /* latest requirement version */ },
        "assessment": { /* latest assessment version */ },
        "resource": { /* ... */ },
        "dev": { /* ... */ }
      }
    }
  ]
}
```

### 2.2 版本展示元数据

**当前前端做法：**

```tsx
const projectName = version.payload?.basicInfo?.projectName
```

### 2.3 PM 工作台总览

**当前问题：** 前端需要发 4-5 个请求

**建议后端提供：**
```
GET /pm/workbench?versionId=...
```

```json

  "narrative": { /* narrative */ },
  "deliverables": [ /* deliverables */ ],

  "seal": { /* seal record */ }
```

## 3. 不建议前端直接绑定的字段（🔴 风险）

### 3.1 assessment_versions.payload

| 字段 | 风险 | 建议 |
|-----|------|------|
| `payload._v1Type` | 内部字段，未来移除 | 不绑定 |
| `payload._v1*` | 全部内部保留字段 | 不绑定 |
| `payload.basicInfo` | 仅 requirementImport 类型有 | 防御性读取 |
| `payload.selectedTemplateItems` | 仅 assessment 类型有 | 防御性读取 |
| `payload.rows` | resource/dev 类型结构不同 | 通过 Service 层统一 |

### 3.2 软引用字段

| 字段 | 风险 | 建议 |
|-----|------|------|
| `linkedSowId` | 可能为 null，未来补 FK | 不直接展示 |
| `linkedNarrativeId` | 当前未同步，可能 stale | 不直接展示 |
| `parentVersionId` | 仅用于版本血缘 | 不直接展示 |

### 3.3 内部状态字段

| 字段 | 风险 | 建议 |
|-----|------|------|
| `ownerUserId` | UUID，无可读性 | 通过 Service 层关联 username |
| `checkedOutByUserId` | UUID，无可读性 | 通过 Service 层关联 username |

## 4. 接口稳定性说明

| 接口类别 | 稳定性 | 演进计划 |
|---------|--------|---------|
| Auth / Users | 🟢 高 | 基本冻结 |
| History Projects | 🟢 高 | 可能补 `projectName` 等字段 |
| PM Workbench 个体服务 | 🟢 高 | 基本冻结 |
| PM Workbench 复合接口 | 🔴 低 | 待设计（workbench / grouped / display-meta） |
| Versions | 🟡 中 | payload 结构可能随业务演进 |
| Change Submissions | 🟡 中 | diffResult / newEstimate 结构未定型 |
| AI 评估 | 🔴 低 | 模型/提示词持续迭代 |

## 5. 前端使用建议

### 5.1 推荐模式

```tsx
// ✅ 推荐：通过统一 Service 消费
const workbenchData = await pmWorkbenchService.getWorkbench(versionId);

// ✅ 推荐：通过 display-meta 获取展示字段
const meta = await versionService.getDisplayMeta(versionId);
```

### 5.2 避免模式

```tsx
// ❌ 避免：直接读取 payload 深层字段
const name = version.payload?.basicInfo?.projectName;

// ❌ 避免：直接拼接用户 ID
const userName = version.ownerUserId; // 这是 UUID，不是人名

// ❌ 避免：client-side 做大量分组聚合
const groups = versions.reduce((acc, v) => { /* 60 行逻辑 */ }, {});
```

### 5.3 错误处理建议

| 场景 | 建议 |
|-----|------|
| `payload` 字段缺失 | 提供兜底文案，不抛错 |
| `versionCode` 重复 | 后端已去重，前端无需处理 |
| `status` 为未知值 | 走默认标签（如"未知状态"） |
| `projectName` 为空 | 显示"未命名项目" + versionCode |
