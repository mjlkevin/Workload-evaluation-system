## 三、核心资产详细分析

### 3.1 config/auth/users.json（~1.3KB，4 用户）

```json
{
  "users": [
    {
      "id": "uuid",
      "username": "mjlkevin",
      "passwordHash": "bcrypt...",
      "role": "admin",
      "status": "active",
      "createdAt": "ISO8601",
      "lastLoginAt": "ISO8601"
    }
  ]
}
```

- **前端消费方**：`ui/V0_SAAS/lib/workload-service.ts` → `/api/v1/auth/me`、`/api/v1/auth/users`
- **v2 现状**：无 PG users 表，仍用 JSON store (`src/middleware/auth.ts` 中的 `loadUsersStore`)
- **风险点**：密码哈希格式需保持兼容；role 字段 v1 用 `admin/sub_admin/user`，v2 RBAC 已升级为 V2Role 映射
- **迁移方式**：直接逐条插入；密码哈希保持原样

### 3.2 config/auth/invite-codes.json（~800B，3 条）

- **前端消费方**：后台管理页 `/api/v1/auth/invite-codes`
- **v2 现状**：无 PG 表，用 `auth.repository.ts` 读写 JSON
- **迁移方式**：量小，直接逐条写入新表或保持 JSON

### 3.3 config/versions/records.json（~1.4MB，大量记录）

这是**最大、最核心的历史资产**。结构：

```json
{
  "records": [
    {
      "id": "uuid",
      "type": "requirementImport | assessment | resource | dev | global",
      "versionCode": "RI-20260328001A",
      "templateId": "tmpl-import-1774340190465",
      "ownerUserId": "uuid",
      "status": "draft | reviewed | published | archived",
      "payload": { ... },
      "createdAt": "ISO8601",
      "updatedAt": "ISO8601",
      "createdByUserId": "uuid",
      "createdByUsername": "...",
      "updatedByUserId": "uuid",
      "updatedByUsername": "...",
      "checkoutStatus": "checked_in | checked_out",
      "versionDocStatus": "drafting | reviewed",
      "majorLetter": "A",
      "minorNumber": 1,
      "baseCode": "RI-20260328001A",
      "isHistoricalArchive": false
    }
  ]
}
```

- **前端消费方**：
  - `version-history-dialog.tsx` → 展示版本历史列表
  - `version-vcs-toolbar.tsx` → checkout/checkin/promote 操作
  - `workload-service.ts` → `listModuleVersions`、`listGlobalVersions`、`checkoutVersionById` 等
- **v2 现状**：已有 `assessment_versions` PG 表，字段高度兼容
- **关键映射**：v1 `records` → v2 `assessment_versions`，`payload` 字段可直接整体放入 jsonb
- **风险点**：
  - v1 `type` 含 `"global"`，v2 `assessment_versions` 没有 `type` 字段（通过 `versionCode` 前缀区分）
  - v1 `status` 有 `"published"`，v2 enum 是 `draft|checked_out|checked_in|promoted|sealed`，需状态映射
  - v1 的 `payload` 结构与 v2 的 `payload` 不完全一致，需验证

### 3.4 config/versions/version-code-rules.json（~2KB，6 条规则）

```json
{
  "rules": [
    {
      "id": "rule-global",
      "moduleKey": "global",
      "moduleName": "总方案",
      "moduleCode": "GL",
      "prefix": "GL",
      "format": "{PREFIX}{MM}{NNN}",
      "sample": "GL04001",
      "status": "active",
      "effectiveAt": "ISO8601",
      "updatedAt": "ISO8601"
    }
  ]
}
```

- **前端消费方**：`workload-service.ts` → `getActiveRuleSet()`
- **v2 现状**：无 PG 表，由 `system.routes.ts` 提供 API，数据来自 JSON
- **迁移方式**：量小，结构简单，可直接建表或保持 JSON

### 3.5 config/templates/example-template.json（~414KB）

- **前端消费方**：`workload-service.ts` → `listTemplateSummaries()`、`getTemplateDetail()`
- **v2 现状**：无 PG 表，由 `templates.usecase.ts` 直接读写 JSON
- **结构特点**：含大量 `groups` 和 `items`，是 Excel 解析后的结构化数据
- **迁移风险**：体积大，字段多，需确认哪些字段是前端真正消费的

### 3.6 config/teams/store.json（~30KB）

- **前端消费方**：`workload-service.ts` → 团队管理、评审、评论
- **v2 现状**：无 PG 表，由 `team.usecase.ts` 读写 JSON，含乐观锁版本号机制
- **结构**：`{ version: number, teams: [...], auditLogs: [...] }`
- **迁移风险**：`version` 字段用于乐观锁并发控制，PG 迁移后需重新设计并发策略

### 3.7 config/system/*.json（配置类）

- `implementation-dependency-rules.json`：实施依赖规则，DSL 引擎消费
- `requirement-settings.json`：需求系统配置，含 **Kimi API Key**（敏感信息！）
- **建议**：配置类资产可保持 JSON 或迁入 `system_config` 通用配置表；**API Key 严禁迁入版本控制**

## 五、结论

**P0 必须迁（4 项）**：users、invite-codes、versions/records、version-code-rules
**P1 建议迁（3 项）**：templates、teams、system 配置
**P2 不迁（2 项）**：mock 数据、示例规则

最大风险点：`records.json` 的 `payload` 兼容性和 `status` 状态映射。
